# 台股智選：AI 開發規則

本檔適用於整個儲存庫。內容只保存長期且穩定的工作規則；分支、版本、網址、部署狀態與近期架構現況應記錄在 `PROJECT_CONTEXT.md`。若兩者衝突，以本檔的安全與品質規則優先，並先查證實際程式碼、資料庫及正式環境狀態。

## 職責與必要閱讀

你負責本專案的系統架構、全端開發、資料工程、量化系統與品質驗證。

開始修改前，依序閱讀：

1. `PROJECT_CONTEXT.md`（若存在）。
2. `README.md`。
3. 本次任務說明。
4. 與任務相關的現有程式碼、API、資料表、migration、文件與測試。
5. `package.json` 中實際存在的指令，以及相關部署或維護 Runbook。

若本次需求使用 `TASK_TEMPLATE.md`，以其中已填寫的任務範圍與驗收條件作為本次工作邊界；未填寫的範例文字不構成需求。

文件與實作不一致時，不得自行猜測；應以可驗證的程式碼、schema、API 回應及環境狀態確認差異並回報。

## 工作原則

1. 先理解現有架構、資料流與既有測試，再開始修改。
2. 優先採用最小必要修改，不任意重寫整個專案。
3. 保留目前正常運作的功能與既有 API 相容性。
4. 不得捏造檔案、欄位、函式、資料表、API 回應、測試結果、Commit、部署或正式環境狀態。
5. 發現需求衝突時，依序優先考慮：
   - 資料正確性
   - 權限與安全性
   - 系統穩定性
   - 手機操作體驗
   - 載入效能
   - 視覺細節
6. 未經明確要求，不得進行大規模重構、框架更換或資料庫破壞性操作。
7. 未經明確要求，不得執行正式環境部署、Git push、正式資料庫 migration、正式資料回填或其他會改變外部狀態的操作。
8. 保留使用者既有及未提交的修改；不得覆蓋、還原或刪除不屬於本次任務的變更。
9. 不得使用 `git reset --hard`、破壞性 checkout、強制推送或其他可能遺失資料的操作，除非使用者明確授權並已確認影響範圍。

## 維護模式與正式環境

1. 純閱讀、分析、查詢及只修改本機檔案時，不切換正式網站維護模式。
2. 正式部署、正式 migration、資料回填、直接寫入正式資料或會造成混合版本的操作前，必須先依 `docs/V20-DELIVERY.md` 切換維護模式並暫停相關排程。
3. 使用專案既有狀態機，不得自行繞過：

```bash
npm run maintenance:status
npm run maintenance:enter -- --confirm
npm run maintenance:verify -- --confirm
npm run maintenance:signature -- verify
npm run maintenance:reclose -- --confirm
npm run maintenance:resume -- --confirm
```

4. `enter` 後必須確認網站及排程確實進入維護狀態，才能進行正式變更。
5. `verify` 僅進入受控驗證階段；驗證失敗必須執行 `reclose`。全部正式驗證通過後，才能執行 `resume` 恢復排程並開站。
6. 缺少正式環境憑證或無法確認維護狀態時，停止正式變更並如實回報，不得聲稱已開啟或解除維護模式。
7. 維護模式不會提升本機建置或測試速度，不得為純本機工作製造不必要的正式服務中斷。

## 手機與介面要求

本專案主要服務 iPhone Safari、手機直式 PWA 與單手操作情境。修改介面時必須檢查：

- 是否出現非預期水平捲動。
- 底部導覽及安全區是否遮擋內容。
- 主要點擊區是否適合觸控，原則上至少為 44 × 44 CSS px。
- 文字是否過小、截斷、重疊或隱藏關鍵風險資訊。
- 載入、無資料、部分資料、快取資料、錯誤及重試狀態是否清楚且可區分。
- 資料日期、取得時間、狀態、來源與是否即時是否完整可見。
- 手機與桌面版是否都能正常使用。
- 實際 iPhone 尺寸或等效 Playwright viewport 是否通過回歸驗證。

## 資料規則

1. 官方資料優先於第三方資料。
2. API 空值、逾時、錯誤或未取得資料不得顯示為數值 `0`。
3. 必須明確區分：
   - 實際數值為 0
   - 尚未取得
   - 來源未提供
   - 歷史資料不足
   - API 錯誤
   - 配額等待中
   - 使用舊快照備援
4. 重要資料必須保留資料日期、實際發布時間、系統取得時間、來源名稱、修訂狀態及模型或資料版本。
5. 不同來源的日期、`Asia/Taipei` 時區、單位、交易日與資料頻率必須統一；程式執行時間不得冒充交易日期。
6. 舊快照只能作為有明確標示的備援，不能冒充最新資料。
7. 不得使用未來資料重新計算或污染過去的推薦、校準與回測。
8. 已發布的推薦快照不得直接修改；修正時必須建立新的 revision 或發布批次，並保留原始版本。
9. 市場別與商品類型是不同維度：以 `market` 區分 TWSE／TPEx，以 `instrument_type` 區分股票／ETF／ETN 等，不得把「上市、上櫃、ETF」視為互斥分類。
10. 短波段與中期模型不得混用分數、期間、勝率、樣本或風險定義。
11. 因子資料缺漏時，必須由硬性閘門排除，或依已記錄且可重現的規則重新正規化權重；不得默默補零後繼續計分。

## 模型規則

1. 大型語言模型不得直接決定正式排名、勝率或買賣分數，只能負責資料整理與可驗證的文字解釋。
2. 所有模型權重必須對應實際有效資料。
3. 不得保留權重卻將輸入固定寫成 0。
4. 原始分數、成本後分數與風險調整分數必須明確區分。
5. 畫面上的分數名稱、尺度、範圍與排序欄位必須和實際計算一致。
6. 樣本不足或尚未完成 Walk-forward／樣本外校準時，不得顯示推估勝率或期望報酬。
7. 模型版本、資料版本、發布批次、revision 及內容雜湊必須可追溯。
8. Challenger 模型不得未經完整樣本外驗證直接取代正式 Champion 模型。
9. 回測與驗證必須包含成本後報酬、超額報酬、回撤、樣本數及市場狀態，不得只使用命中率判斷模型品質。

## API 規則

API 呼叫應依端點用途具備適用的 timeout、有限重試、指數退避與 jitter、速率限制、快取、錯誤分類、last-good snapshot 及 circuit breaker。

另外必須遵守：

1. 修改 API 前先確認實際回傳格式，不得猜測欄位。
2. HTTP 200 不代表資料內容正確、完整或日期已更新；必須驗證 schema、狀態及資料日期。
3. 400、401、402、403 等參數、權限或憑證錯誤不得無限重試。
4. 429、逾時與 5xx 只能依規則有限重試；非冪等寫入不得在沒有 idempotency 保護時自動重送。
5. 不得把專案內部節流規則描述成官方 API 配額。
6. 錯誤不得被吞掉或改寫成成功狀態；前端必須收到可分類但不洩漏敏感內容的錯誤資訊。

## 敏感資料

1. API Key、Token、Cookie、密碼、Supabase service role、私鑰及其他憑證只能存放於 Vercel 伺服器環境變數、Supabase Vault 或其他核准的伺服器端秘密儲存空間。
2. 不得將敏感資料寫入前端、Git、`AGENTS.md`、`PROJECT_CONTEXT.md`、測試 fixture、日誌、截圖或回覆；Base64 等編碼字串仍視為敏感資料。
3. 顯示命令或日誌前必須遮罩秘密，只保留辨識所需的最少字元。
4. 已出現在聊天、Commit、日誌或公開位置的金鑰一律視為可能外洩，應撤銷並重新簽發，不得繼續沿用。
5. `.env.example` 只能放變數名稱與無效範例值，不得放真實憑證。

## 資料庫與權限

資料庫目前分為：

- CORE：登入、帳戶、自選股與使用者資料。
- MARKET：市場資料、模型、排行榜、回測、發布快照與管理功能。

必須遵守：

1. CORE migration 不得套用到 MARKET。
2. MARKET migration 不得套用到 CORE。
3. 新增或修改資料表時必須檢查 RLS、GRANT、RPC、索引及 service-role 邊界。
4. 管理員功能必須同時受到前端、後端、RLS 與 RPC 保護。
5. 隱藏前端按鈕不等於完成權限保護。
6. 一般使用者不得直接呼叫管理員 API 或讀取內部資料表。
7. 不得破壞既有資料或直接刪除正式資料表。
8. 資料庫變更必須提供完整 migration，以及可行的 rollback、roll-forward 或備份還原方案；不可逆操作必須在執行前明確揭露。
9. 套用 migration 前必須確認目標專案、遠端 migration history、transaction dry-run 結果及維護狀態。

## 快取與競態

修改時必須檢查：

- 舊請求是否可能覆蓋新的篩選結果。
- 是否需要 AbortController、request ID、publication key 或 query signature。
- LocalStorage 是否具備 TTL、schema 版本、使用者／帳戶隔離及登出清理。
- Service Worker 是否可能長期回傳舊 API 資料，或刪除不屬於本應用的快取。
- API schema 改變後是否同步提高快取版本並處理舊資料遷移。
- 登入、登出或切換帳戶後是否殘留其他使用者資料。
- 自選股是否綁定正確的 run ID、publication key、content hash 與 data date。
- 自選股數量較多時是否能完整分批載入。
- 是否產生重複請求、無限迴圈、過期 promise 或未清除的事件監聽器。

## Git 與變更範圍

1. 修改前先檢查工作目錄狀態；將既有變更視為使用者所有。
2. 只修改本次任務必要的檔案，不順手格式化或重寫無關檔案。
3. 不得以刪除、reset 或重新產生整個目錄的方式避開衝突。
4. Commit、push、建立 PR、合併、部署與發布都需要符合使用者本次明確授權的範圍。
5. Commit 前不得只看整體 diff；必須逐一確認本次變更及是否意外包含秘密或產物。
6. 一般交付直接提交並上傳已授權的工作分支，不建立 ZIP；只有使用者明確要求離線封裝時才產生壓縮檔。
7. 上傳原始碼不代表已授權合併 PR、部署正式環境、套用 migration 或改寫正式資料。

## 測試要求

1. 先讀取 `package.json`，只執行實際存在且與修改範圍相符的指令。
2. 優先執行最小相關測試；跨模組、高風險、資料庫、快取、權限或正式發布變更，再執行完整回歸。
3. `npm ci` 只在乾淨／新環境、依賴變更或 lockfile 需要驗證時執行。
4. `npm audit` 只用於依賴或安全稽核任務；它與專案的 `npm run audit` 正式站稽核不是同一項操作。
5. 純文件修改不需要執行完整程式測試，但必須檢查 Markdown、指令名稱、路徑及內容一致性。
6. 一般程式修改至少執行適用的：

```bash
npm run build
npm run validate
# 與修改範圍直接相關的 test:* 指令
```

7. 高風險或發布前依 `docs/V20-DELIVERY.md` 執行適用測試，最後以 `npm test` 做完整回歸；需要實際瀏覽器流程時執行 `npm run test:e2e`，正式站稽核只在獲得授權且環境正確時執行 `npm run audit`。
8. 測試若失敗、逾時、被中止或未執行，不得宣告通過。必須回報確切命令、退出狀態、失敗原因與未完成項目。
9. 不得為了讓測試通過而降低斷言、跳過高風險案例或更新基準產物，除非已證明新結果正確並說明原因。

## 完成條件與回報格式

1. 執行期間的進度更新使用：`完成 X%｜預估剩餘 Y 分鐘`。百分比與時間必須誠實；無法精確估計時使用範圍。
2. 簡短且立即完成的回答可只在最終回覆顯示：`完成 100%｜剩餘時間 0 分鐘`。
3. 最終回覆必須列出：
   - 完成的變更
   - 修改的檔案
   - 實際執行的測試及結果
   - 未執行或未完成的驗證
   - 是否 Commit、push、部署或修改正式資料
   - 若涉及正式環境，目前維護模式與服務狀態
4. 沒有實際驗證的部署、資料日期、API、權限或畫面狀態，必須清楚標示為未驗證。
5. 只有在要求的修改已完成、適用測試通過且沒有隱藏阻塞時，才能宣告任務完成。
6. 不得宣稱「專案完全沒有 Bug」。可使用的結論是：「在已執行的測試範圍與環境中，未發現尚未修復的已知或可重現問題。」
